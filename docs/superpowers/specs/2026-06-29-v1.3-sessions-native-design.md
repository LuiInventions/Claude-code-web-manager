# v1.3.0 — Native Sessions, Activities, Launcher Fixes & Provider Check

**Date:** 2026-06-29
**Branch:** `feat/v1.3-sessions-native`
**Status:** Approved (design), ready for implementation

## Goal

Ship Claude Code Control Center **v1.3.0**: rebuild the Sessions tab so the
Pixel-Office and Flow-Graph use the *native* designs of the reference projects
([pixel-agents](https://github.com/pixel-agents-hq/pixel-agents),
[agent-flow](https://github.com/patoles/agent-flow)) — a real canvas office room
and a real force-directed hex graph — driven by **richer live activity** and
**subagents** parsed from each session. Plus four targeted fixes (folder
outlines, an xterm layout bug, a "close all sessions" button, the Groq/provider
403) and a release.

Two approved decisions:
1. **Render natively from scratch on `<canvas>`** (no upstream assets/engine);
   add only `d3-force`. Self-contained/offline.
2. **Parse each PTY output buffer server-side** to derive rich activity
   (`thinking`/`working`/`waiting`≈needs-approval/`done`/`error`) and real
   in-session subagents.

## Non-goals

- No switch of the launcher away from interactive PTY to stream-json.
- No vendoring of upstream sprite assets or canvas engines.
- We cannot execute the user's private (encrypted) provider keys from the dev
  session; the deliverable for "verify all providers" is a self-serve in-app
  **Test** button + the Groq default fix + clearer errors.

## Foundation — activity & subagent detection

The launcher spawns `claude --dangerously-skip-permissions` in a real PTY
(`lib/server/claude-pty.ts`). The full ANSI scrollback is buffered per session.
Because `--skip-permissions` is set, Claude rarely shows a literal permission
prompt; the meaningful "needs attention" state is **the agent finished its turn
and is waiting for the user to type** (input box shown, output idle).

### `lib/session-activity.ts` (new, pure, unit-tested)

```ts
export type LiveActivity =
  | "thinking" | "working" | "waiting" | "done" | "error";

export interface DetectedSubagent {
  id: string;          // stable per (sessionId + ordinal)
  label: string;       // e.g. Task description / agent type
  activity: LiveActivity;
}

export interface ActivitySignal {
  activity: LiveActivity;
  subagents: DetectedSubagent[];
}

export function detectActivity(input: {
  tail: string;        // normalizeConsoleText() output (ANSI stripped)
  status: "running" | "done" | "error";
  lastDataAtMs: number;
  now: number;
}): ActivitySignal;
```

Heuristics (over the normalized tail; reuse `stripAnsi`/`normalizeConsoleText`
from `console-read.ts`):
- `status !== "running"` → `done` / `error` (exit code wins).
- Spinner / `esc to interrupt` / `✻` / `Thinking…` near the end → `thinking`.
- Active tool lines (`●`, `⎿`, `Bash(`, `Edit(`, `Write(`, `Read(`, `Search(`,
  `Update(`) near the end → `working`.
- Output idle (`now - lastDataAtMs` > ~1200ms) **and** an input box / prompt
  glyph (`│ >`, `╰`, `> `) visible at the tail → `waiting`.
- Otherwise → `working` (default while running).
- Subagents: scan for `Task(` / `● Task` dispatch markers (and their `⎿`
  follow-ups); each becomes a `DetectedSubagent` with its own activity
  (working while its block is the latest, done once a result line follows).

This is explicitly heuristic and documented as such.

### `lib/server/claude-pty.ts`

- Track `lastDataAt` per session (set in `term.onData`).
- `PtySessionInfo` gains: `activity: LiveActivity`, `subagents: DetectedSubagent[]`,
  `lastActivityAt: number`.
- `listPtySessions()` calls `detectActivity()` per session (cheap; tail only).

### `lib/sessions.ts`

- `VisualSession` gains `activity?: LiveActivity` and `subagents?: DetectedSubagent[]`.
- New `sessionActivity(s): LiveActivity` prefers `s.activity`, falls back to
  `statusActivity(s.status)` mapped into the richer enum.
- Color/`hashId` helpers unchanged and reused for both canvas views.

`/api/launcher/live-sessions` passes the new fields through unchanged (it already
returns `listPtySessions()`).

## 1+2. Sessions tab — native canvas views

`SessionsSection.tsx` keeps the pixel/flow toggle, footer credits, refresh and
empty state. Polling interval tightened to ~1000ms. Passes `VisualSession[]`
(now incl. `activity`+`subagents`) to whichever view is active.

### `PixelOfficeView.tsx` (rewritten — canvas)
- One DPR-aware `<canvas>` drawing a **single shared office room**: tiled floor,
  back wall, and a grid of desks. One pixel-sprite character per session seated
  at a desk; subagents seated as smaller companions beside their parent.
- Sprites drawn programmatically (filled rects, crisp), deterministic per
  `sessionColor(id)`. `requestAnimationFrame` loop.
- Activity → animation: `working`=typing hands, `thinking`=idle bob + thought
  dots, `waiting`=turn to viewer + "!" speech bubble (**the needs-approval
  signal**), `done`=calm + ✓ bubble, `error`=red shake + "!".
- Hover/click a character → floating panel (project, prompt, model, status,
  subagent count). New characters fade/spawn in; removed ones fade out.
- Resilient: rAF paused when tab hidden; redraws on resize (ResizeObserver).

### `FlowGraphView.tsx` (rewritten — canvas + d3-force)
- DPR-aware `<canvas>` + `d3-force` simulation. Nodes: root **Launcher** →
  session **hex** nodes → subagent child hexes; KI-Modus batches route through a
  batch-hub node. Edges: tapered bezier "beams"; particles travel along edges of
  active (working/thinking) sessions.
- Node visuals by activity: glow color (thinking=cyan pulse, working=blue,
  waiting=amber radar ring, done=green, error=red), breathing scale, index badge,
  project label, status dot.
- Pointer pan + wheel zoom, auto-fit on data change. Click a node → same details
  panel as the office.
- Forces: many-body charge, link distance, center, collision; `alphaDecay`
  tuned; simulation cooled when idle.

New dependency: `d3-force` + `@types/d3-force`.

## 4. Launcher xterm layout bug

Root cause: `fit.fit()` runs synchronously right after `term.open(el)` in
`ClaudeCmdPane.tsx`, before the sidebar-collapse + grid layout settle; the
`ResizeObserver` is attached afterward, so the wrong initial fit can stick until
a later sidebar toggle resizes the container.

Fix:
- `ClaudeCmdPane.tsx`: replace the synchronous initial `fit.fit()` with a
  deferred fit (double `requestAnimationFrame`), refit on `ws.onopen`, and a
  couple of short delayed refits (≈60ms, 250ms). Keep a `doFit` ref. Add an
  optional `layoutNonce?: number` prop; a `useEffect([layoutNonce])` schedules a
  `rAF` refit.
- `LauncherSection.tsx`: hold a `layoutNonce` that increments when `sidebarOpen`
  changes and when `activeBatchId` changes; pass it to every `ClaudeCmdPane`.
  Deterministically heals "shifted on start" and "shifted after switching pages".

## 5. Launcher — "Close all sessions"

In the boxes bar (`{visibleCount} boxes`), add a right-aligned (`ml-auto`)
"Close all" button (Trash/X icon). Click → `window.confirm` naming the **total**
`sessions.length` → `DELETE /api/launcher/live-sessions?id=…` for every session →
clear `sessions`, `selectedId`, reviews stay; reopen the sidebar. Generalized
from `closeSession`.

## 6. Providers / Groq 403 + verify-all

- **Groq default model**: `openai/gpt-oss-20b` is gated → 403. Change Groq's
  `defaultModel` to `llama-3.3-70b-versatile` and reorder `models` so a
  generally-available model leads. (`lib/providers.ts`; update
  `lib/__tests__/providers.test.ts` if it asserts the default.)
- **Per-provider Test**: new `POST /api/providers/test` → minimal
  `chat.completions.create({ max_tokens: 1 })` with the saved key+model →
  `{ ok, status?, error? }` (never leaks the key). `SettingsSection.tsx` gains a
  **"Verbindung testen"** button under the AI provider card showing OK / the real
  error (so the Groq 403 reason is visible, and every provider is checkable).
- **Clearer errors**: improve/split/review surface provider label + model +
  status in the thrown message.

## Risks

- Activity/subagent heuristics depend on Claude Code's TUI output and may need
  tuning; they degrade gracefully to `working`/coarse status.
- Local `electron:build` may be environment-heavy; the authoritative artifact is
  the CI `v1.3.0` tag build.
- d3-force adds a small dependency; acceptable and offline-bundled.
